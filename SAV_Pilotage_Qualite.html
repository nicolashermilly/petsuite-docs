Process complet — fin de sprint (ordre à respecter)
1
5 min avant la fin
Lancer l'analyse qualité
Exécuter le script Python depuis le dossier racine du projet. Lecture seule — produit un rapport et identifie les patches disponibles.
cd "D:\...\micro_logiciel_frontend_nextjs"
python animgest_quality.py --sprint S58
🐍 animgest_quality.py
2
5 min — fin de sprint
Lancer le diagnostic maintenance
Vérifie les conteneurs Docker, les 10 routes API critiques, les fichiers .bak et les credentials exposés. Génère maintenance_report.json.
python animgest_maintenance.py --sprint S58
🐍 animgest_maintenance.py
3
Optionnel — patches non-risqués
Appliquer les patches qualité validés
Relancer en mode --apply. Chaque patch est présenté avec son niveau de risque. Un backup automatique est créé avant toute modification.
python animgest_quality.py --apply --sprint S58
Auto-applicables sans risque : JSDoc, src/services/. Manuels : refacto routes, window.open() → Modal, SQL commentaires.
4
5-10 min — obligatoire

Ouvrir le générateur, parcourir les 7 onglets. En fin de génération, un bloc de scores apparaît automatiquement avec un bouton pour ouvrir le Dashboard Qualité avec les données pré-remplies.

Règles qualité — ne jamais enfreindre
Règles architecture
  • cd deploy obligatoire avant tout docker compose
  • COPIER_TOUTES_PAGES_v2.ps1 avant build frontend
  • app.use() jamais app.get() — nginx strips /api/
  • getDb() uniquement — jamais createConnection
  • build --no-cache puis up -d — jamais restart
  • restart nginx après chaque rebuild
Règles code
  • PDF / éditions : Modal uniquement — jamais window.open()
  • "use client" : ligne 1 absolue de tout fichier TSX
  • Modifications TSX : via scripts Python (apostrophes FR)
  • Pas de backtick dans les strings SQL
  • inject_v4_final.js : ne jamais utiliser
  • CAC : accessible uniquement depuis fiche séance — pas de lien sidebar
Règles BDD
  • FK : Relation NomTable - description
  • Champ ordinaire : description de la valeur attendue
  • Colonnes sans DEFAULT : NOW() explicite dans INSERT
  • ENUM migration : VARCHAR → UPDATE → ENUM (3 étapes)
  • ALTER CHANGE : ne pas reconstruire la clause DEFAULT
  • Frimousse ANI_ID=22 : ne jamais supprimer
Règles Git & docs
  • git pull --rebase avant tout push sur petsuite-docs
  • Délai déploiement GitHub Pages : 3-5 min après push
  • Backup propre : C:\Backup\micro_logiciel\20260315_0002
  • Backups à partir de 20260315_0042 : code corrompu — ne pas utiliser
  • Script sauvegarde v7 : lancer en fin de chaque session

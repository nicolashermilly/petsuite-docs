21 mai 2026 · Multi-tenant phase 1 : login tenant + AsyncLocalStorage

✅ Livraisons principales

  • MULTI-71-01 backend + frontend — Authentification deplacee dans BDD tenant. Nouvelle route publique /t/[tid]/login rendant le formulaire de connexion specifique a chaque tenant (resolution dynamique du tid via segment URL).
  • MULTI-71-02 tenant-aware DB access — Mise en place d'AsyncLocalStorage Node.js pour propager le contexte tenant a travers toutes les couches d'execution (middleware nginx → resolver → service → pool MariaDB tenant approprie). Plus de fuite cross-tenant possible.
  • 7 tenants actifs operationnels — t0001/43/44/47/48/49/50/51 tous joignables via /t/[tid]/login avec bases isolees.

🔧 Architecture multi-tenant phase 1

  • Deux pools MariaDB distincts : mariadb_ops (port 3330, base centrale + licensing) et mariadb_tenants (port 3331, bases isolees par tenant).
  • Resolver tenant injecte dans le contexte avant chaque requete, accessible via getCurrentTenant() n'importe ou dans le code applicatif.
  • Provisioning automatique d'une BDD tenant depuis le template animgest_template lors de l'activation d'une cle licensing JWT.

📝 Dette technique soldee

  • D-S115-J3-1 (route /t/[tid]/login 404) resolu implicitement par MULTI-71-01.
  • D-S115-J2-1 (nginx X-Forwarded-For sur HTTPS) livre en pre-sprint.

1 — Animaux & Races
ANI-01 Numéro de puce — unicité & format 15 chiffres
Le numéro de puce ANI_PUC est la clé universelle de l'animal. Il doit contenir exactement 15 chiffres, sans espace ni tiret. Il est unique dans toute la base et non modifiable après création.
UNIQUE KEY uq_ani_puc CHECK REGEXP ^[0-9]{15}$ Frontend : maxLength=15, required
-- Cas de recette : INSERT INTO animal (ANI_PUC, ...) VALUES ('123', ...); -- ✗ Rejeté (< 15 chiffres) INSERT INTO animal (ANI_PUC, ...) VALUES ('1234A6789012345', ...); -- ✗ Rejeté (lettres) INSERT INTO animal (ANI_PUC, ...) VALUES ('250269811400022', ...); -- ✓ Accepté INSERT INTO animal (ANI_PUC, ...) VALUES ('250269811400022', ...); -- ✗ Rejeté (doublon)
ANI-02 Propriétaire obligatoire — lien tiers client
Tout animal doit être rattaché à un tiers de type CLIENT ou CLIENT_FOURNISSEUR. La FK ANI_CLI → tiers.TIE_ID est ON DELETE RESTRICT : impossible de supprimer un client qui a des animaux.
FK fk_animal_client ON DELETE RESTRICT Frontend : select required
ANI-03 Races — 1 principale + 1 secondaire optionnelle
L'animal peut avoir jusqu'à 2 races (ANI_ESP_1 et ANI_ESP_2). Les deux sont optionnelles. Si la race est supprimée du référentiel, les champs passent à NULL (ON DELETE SET NULL).
FK ON DELETE SET NULL 392 races disponibles · 8 espèces
ANI-04 14 critères CAC — cases à cocher booléennes
Les colonnes ANI_CAC_1 à ANI_CAC_14 sont des TINYINT(1) avec valeur par défaut 0. Le score CAC est calculé à chaque séance de suivi, pas directement sur la fiche animal.
DEFAULT 0 · Mis à jour via PUT /animaux/:id
2 — Séances & Agenda
SEA-01 Unicité EDC par animal — trigger BDD
Une seule séance de type EDC (Étude de comportement) peut exister par animal, hors statut ANNULÉE. Géré par 2 triggers (BEFORE INSERT et BEFORE UPDATE). L'animal est obligatoire pour ce type de séance.
trg_sea_unique_par_animal_ins trg_sea_unique_par_animal_upd STY_UNIQUE_PAR_ANIMAL = 1
-- Erreurs levées : SQLSTATE 45000 : 'Une séance de ce type existe déjà pour cet animal' SQLSTATE 45000 : 'Type de séance unique par animal : ANI_ID obligatoire'
Créer une 2ème EDC pour le même animal → message d'erreur attendu
L'erreur doit s'afficher clairement côté frontend
Annuler l'EDC existante → recréer une nouvelle EDC pour le même animal → doit être accepté
Créer une EDC sans sélectionner d'animal → rejet BDD + message frontend
SEA-02 Calcul automatique des totaux de séance — 3 triggers AFTER
À chaque INSERT / UPDATE / DELETE d'une ligne de séance (seance_ligne), les totaux HT/TVA/TTC de la séance parente sont automatiquement recalculés et mis à jour via les triggers trg_sel_after_*. Le champ SEA_MODIFIE_LE est également mis à jour.
trg_sel_after_insert trg_sel_after_update trg_sel_after_delete
Ajouter une ligne prestation → vérifier que SEA_MONTANT_TTC se met à jour immédiatement
Modifier la quantité d'une ligne → recalcul automatique des totaux
Supprimer toutes les lignes → SEA_MONTANT_TTC revient à 0
SEA-03 Calcul lignes de séance — arrondi 2 décimales
Les totaux de chaque ligne sont calculés par trigger BEFORE INSERT/UPDATE avec arrondi à 2 décimales : ROUND(QTE * PU_HT, 2) pour le HT, puis TVA appliquée sur le HT calculé.
trg_sel_calc_insert trg_sel_calc_update
SEA-04 Statuts de séance
4 statuts possibles : PLANIFIÉE · RÉALISÉE · ANNULÉE (statut final) · NO_SHOW (statut final). Le statut ANNULÉE est exclu de la règle d'unicité EDC.
SST_FINAL = 1 pour ANNULEE et NO_SHOW
3 — Étude Comportementale (EDC) & Suivi CAC
EDC-01 Une seule fiche EDC par animal
La fiche EDC est unique par animal. Elle est partagée entre toutes les séances EDC de cet animal. Elle comporte 7 sections : Identité · Alimentation · Comportement · Mode de vie · Antécédents · Grille comportementale · Attentes. La grille comportementale est multi-lignes (CRUD).
Unicité garantie par SEA-01 Sauvegarde partielle autorisée
Ouvrir l'EDC depuis une séance → la même fiche s'affiche (pas de doublon)
Sauvegarder partiellement (section 1 seulement) → reprendre depuis section 2 lors d'une séance ultérieure
CAC-01 14 critères CAC par séance — pré-remplissage
Pour chaque séance de suivi, les 14 critères CAC sont cochables individuellement. Lors de l'ouverture d'une nouvelle séance, les critères sont pré-remplis depuis la dernière séance de l'animal. Le score temps réel est calculé côté frontend.
64 libellés paramétrables Score = somme des cases cochées / 14
Créer une 2ème séance de suivi → vérifier que les 14 critères reprennent l'état de la séance précédente
Onglet Évolution → graphique et tableau comparatif mis à jour après sauvegarde
4 — Devis & Contrats
DCE-01 Numérotation automatique DCE-YYMM-NNNN
À l'insertion d'un devis sans numéro, le trigger trg_dce_num génère automatiquement un numéro au format DCE-2603-0001 (YYMM = année 2 chiffres + mois 2 chiffres). Le compteur est par mois via la table dce_compteur_mois.
trg_dce_num BEFORE INSERT Remise à 0 chaque mois
Créer un devis → vérifier le format DCE-YYMM-0001
Créer un 2ème devis le même mois → numéro DCE-YYMM-0002
DCE-02 Calcul automatique des totaux lignes devis
À chaque INSERT/UPDATE de ligne de devis, les totaux sont calculés par trigger : ROUND(QTE × PU_HT, 2) pour le HT, puis TVA sur le HT arrondi, puis TTC = HT + TVA. Le taux TVA est enregistré en snapshot (DCL_TVA_TAUX).
trg_dcl_calc_insert trg_dcl_calc_update Snapshot taux TVA au moment de la création
DCE-03 Procédure sp_dce_creer_depuis_seance — règles strictes
La procédure sp_dce_creer_depuis_seance(p_sea_id, p_forcer) génère un devis depuis une séance avec 3 règles bloquantes :
1. La séance doit exister
2. Si un devis est déjà lié et p_forcer = 0 → erreur
3. Pour une EDC, l'animal est obligatoire
Le devis créé a une validité de 30 jours à partir de la date de séance. Les lignes de séance sont copiées dans le devis.
sp_dce_creer_depuis_seance Validité +30j auto
Générer devis depuis séance EDC sans animal → erreur attendue
Générer un 2ème devis sans forcer → message "Un devis est déjà lié"
Vérifier que la date de validité = date séance + 30j
Vérifier que les lignes de séance sont bien copiées dans le devis
DCE-04 Statuts devis
5 statuts : BROUILLON → ENVOYÉ → ACCEPTÉ ou REFUSÉ ou ANNULÉ. Un devis généré depuis une séance est créé en BROUILLON.
Statuts ENUM BDD
5 — Facturation
FAC-01 Numérotation automatique FAC-YYMM-NNNN
Même mécanisme que les devis — trigger trg_facture_num génère un numéro au format FAC-2603-0001. Compteur mensuel via fac_compteur_mois.
trg_facture_num BEFORE INSERT
Créer une facture → format FAC-YYMM-NNNN vérifié
Séquence des numéros : FAC-2603-0001, FAC-2603-0002… sans saut
FAC-02 Calcul automatique totaux lignes facture
Identique aux devis — triggers trg_fli_calc_insert et trg_fli_calc_update calculent HT · TVA · TTC avec arrondi 2 décimales. Le taux TVA est snapshoté.
trg_fli_calc_insert trg_fli_calc_update
FAC-03 Verrouillage facture après clôture d'exercice
Si l'exercice comptable auquel appartient une facture est clôturé (EXE_CLOTURE = 1), toute modification ou suppression de cette facture est bloquée par trigger. La date de facture est comparée aux bornes de l'exercice.
trg_facture_no_update_if_closed trg_facture_no_delete_if_closed
SQLSTATE 45000 : 'Exercice cloture : modification facture interdite' SQLSTATE 45000 : 'Exercice cloture : suppression facture interdite'
Clôturer l'exercice 2026 → tenter de modifier une facture de 2026 → rejet attendu
Créer une facture sur un exercice non clôturé → modification possible
FAC-04 Statuts facture
4 statuts : BROUILLON → EMISE → PAYÉE ou ANNULÉE. Une facture émise et partiellement réglée apparaît dans l'encours client via la vue v_facture_suivi.
v_facture_suivi · v_encours_client
FAC-05 Client obligatoire — ON DELETE RESTRICT
Une facture doit être rattachée à un tiers existant. La FK est ON DELETE RESTRICT : impossible de supprimer un client qui a des factures.
FK fk_fac_tiers ON DELETE RESTRICT
6 — Règlements
REG-01 Pas de dépassement du montant TTC facturé
Le trigger trg_reglement_check vérifie que la somme des règlements d'une facture ne dépasse pas son montant TTC. Si (déjà payé + nouveau règlement) > TTC, le règlement est bloqué.
trg_reglement_check BEFORE INSERT
SQLSTATE 45000 : 'Montant total des règlements supérieur au TTC'
Facture 100€ → règlement 80€ → règlement 30€ → rejet (80+30 > 100)
Facture 100€ → règlement 60€ → règlement 40€ → accepté (total = 100)
Facture 100€ → règlement 100€ en une fois → accepté
REG-02 Verrouillage règlement après clôture d'exercice
Même que FAC-03 — tout règlement dont la date tombe dans un exercice clôturé est protégé en modification et suppression.
trg_reglement_no_update_if_closed trg_reglement_no_delete_if_closed
7 — FEC & Comptabilité
FEC-01 Génération FEC par exercice — procédure sp_fec_generer_par_exercice
La procédure génère un export FEC complet (journal VT + journal BQ) pour un exercice donné. Chaque ligne reçoit un hash SHA-256 calculé sur tous ses champs via SHA2(CONCAT_WS('|', ...), 256). Un hash global SHA-256 chaîné est ensuite calculé sur toutes les lignes triées.
sp_fec_generer_par_exercice Hash SHA-256 par ligne Hash global chaîné
CALL sp_fec_generer_par_exercice('2026') → vérifier que toutes les factures de 2026 sont présentes
Vérifier l'équilibre : pour chaque EcritureNum, SUM(Debit) = SUM(Credit) via v_fec_export_non_equilibre
Vérifier que FXL_HASH est non NULL sur chaque ligne
Vérifier que FEX_HASH_GLOBAL est non NULL sur l'export
FEC-02 Structure FEC — 3 écritures par facture
Chaque facture génère 3 lignes FEC : (1) Débit 411000 Clients = TTC, (2) Crédit 706000 Prestations de services = HT, (3) Crédit 445710 TVA collectée = TVA (uniquement si TVA ≠ 0). Chaque règlement génère 2 lignes : Débit 512000 Banque + Crédit 411000 Clients.
411000 Clients 706000 Prestations 445710 TVA collectée 512000 Banque
FEC-03 Clôture d'exercice — irréversible
La procédure sp_exercice_cloturer passe EXE_CLOTURE = 1. Cette opération est irréversible — aucune procédure de réouverture n'est prévue. Une fois clôturé, toutes les factures et règlements de la période sont verrouillés.
sp_exercice_cloturer — irréversible
Clôturer l'exercice → confirmer que EXE_CLOTURE = 1 en BDD
Vérifier qu'un message d'avertissement clair est affiché avant confirmation côté frontend
8 — GED Documentaire
GED-01 Validation inter-BDD — objet métier doit exister
Avant de créer un lien document → objet métier, le trigger bi_ged_document_lien_check_objet vérifie que l'objet cible existe dans micro_logiciel. Couvre 13 types : ANIMAL, ARTICLE_PRESTATION, CONTACT, FACTURE, RACE, SEANCE, TIERS, UTILISATEUR, DEVIS_CONTRAT, AVOIR, REGLEMENT, CRM_NOTE, CRM_ACTIVITE.
bi_ged_document_lien_check_objet BEFORE INSERT bu_ged_document_lien_check_objet BEFORE UPDATE 13 types d'objets supportés
SQLSTATE 45000 : 'Objet métier introuvable pour ce lien GED' SQLSTATE 45000 : 'Type d''objet GED (GOT_ID) inconnu' SQLSTATE 45000 : 'Type d''objet métier non pris en charge par la GED'
GED-02 Hash SHA-256 par document — intégrité
Chaque document GED possède un champ DOC_HASH_SHA256 CHAR(64) pour déduplication et contrôle d'intégrité. Le stockage est externe (URI/S3) — pas de BLOB en BDD.
Hash SHA-256 optionnel Stockage URI externe (S3-compatible)
GED-03 Unicité lien (DOC_ID, GOT_ID, OBJ_ID, DLI_ROLE)
Un même document ne peut pas être lié deux fois au même objet métier avec le même rôle. La contrainte uq_dli garantit l'unicité de la combinaison.
UNIQUE KEY uq_dli
GED-04 Statuts document
3 statuts : ACTIF · ARCHIVE · SUPPRIME. La vue v_ged_pj_animal filtre automatiquement les documents SUPPRIMÉS.
v_ged_pj_animal filtre DOC_STATUT ≠ SUPPRIME
9 — CRM
CRM-01 Formulaire lead public — création automatique
Le formulaire accessible via /questionnaire?type=canin|equestre|pension|toiletteur (sans connexion) crée automatiquement un lead dans le CRM. Les champs fixes : Nom · Prénom · Téléphone · Email · Nom animal · Race · Âge · Commentaire. Le consentement RGPD est obligatoire.
GET /api/questionnaires 4 types de questionnaire Consentement RGPD obligatoire
Soumettre formulaire ?type=canin → vérifier création lead dans CRM → Leads
Soumettre sans cocher le consentement RGPD → soumission bloquée
CRM-02 Historique pipeline opportunités
Tout changement de statut d'une opportunité est tracé dans crm_opportunite_histo avec le statut avant/après, la probabilité avant/après, et le montant avant/après.
crm_opportunite_histo
10 — Exercices comptables
EXE-01 Dates cohérentes — CHECK EXE_DATE_DEBUT ≤ EXE_DATE_FIN
La contrainte chk_exe_dates garantit que la date de début d'exercice est antérieure ou égale à la date de fin.
CHECK (EXE_DATE_DEBUT <= EXE_DATE_FIN)
EXE-02 Code exercice unique
Le code exercice (ex: 2026, 2025-2026) est unique. EX2026 est déjà créé en base.
UNIQUE KEY uq_exe_code
11 — Sécurité & Accès
SEC-01 Utilisateur SQL unique — Admin
Un seul utilisateur SQL applicatif : Admin (MariaDB hors Docker, port 3307, --skip-ssl obligatoire). Droits : SELECT, INSERT, UPDATE, DELETE, SHOW VIEW, EXECUTE, TRIGGER, EVENT sur micro_logiciel.*.
Hors Docker · 192.168.1.62:3307 --skip-ssl obligatoire
SEC-02 SIRET — format 14 chiffres
La contrainte chk_tie_siret_format sur la table tiers vérifie que le SIRET, s'il est renseigné, contient exactement 14 chiffres.
CHECK REGEXP ^[0-9]{14}$
SEC-03 Audit log — toutes actions tracées
La table audit_log trace toutes les actions : INSERT / UPDATE / DELETE / LOGIN / EXPORT. Chaque entrée contient la date, l'utilisateur, la table, l'ID objet, un commentaire et un payload JSON optionnel (validé par JSON_VALID).
audit_log CHECK JSON_VALID(AUD_PAYLOAD)

Sprint S114

Anim'Gest - Sprint S114

Portail commercial des licences & provisioning automatique des tenants

Du 12/05/2026 au 18/05/2026 · 7 jours intensifs · Beta release achieved

Vue d'ensemble

Sprint pivotant qui debloque la commercialisation : portail admin pour gerer les licences (3 packs commerciaux + 8 options modulaires) et provisioning automatique de la BDD client en 1 click depuis l'interface.

Au terme de ce sprint, le cycle complet "creation d'un client" passe de plusieurs heures de manipulation manuelle (creation BDD, import schema, configuration) a moins d'une minute en interface admin.

B97 - Portail commercial des licences

Interface admin complete pour le cycle de vie commercial des licences tenants.

Modele commercial

  • 3 packs predefini : Starter, Pro, Enterprise (avec defaults nb_users + options)
  • 8 options modulaires : IA Eliot, OCR, Compta automatique, Multi-tenant, SMS, API externe, Calendly, SMTP premium
  • Mode personnalise : possibilite de modifier les options apres choix d'un pack (badge visuel "Mode personnalise actif")

NouveauListe filtrable des licences

Recherche par email/raison sociale, filtre par statut (Active/Revoquee/Expiree), pagination 20 lignes par page, affichage des options activees sous forme de badges.

NouveauCreation de licence

Modale en 4 sections numerotees : selection pack visuel cliquable, formulaire client (email, raison sociale), parametres (nb_users, metiers, date expiration), toggle options par categorie.

NouveauRevocation avec motif

Action irreversible avec choix du motif (fin contrat, impaye, autre) + detail libre. Trace permanente dans license_revoked.

NouveauRenouvellement

Prolongation de la date d'expiration avec possibilite de changer pack/nb_users/options au passage. Les options actuelles sont conservees par defaut.

B98 - Provisioning automatique des tenants

Automatisation complete de la creation de la BDD client.

Avant / Apres

  • Avant B98 : Creation manuelle de la BDD via HeidiSQL, import du schema via scripts SQL, configuration manuelle des index/triggers/vues. Plusieurs heures de manipulation par client.
  • Apres B98 : 1 click "Provisionner" dans l'interface admin = BDD animgest_tNNNN creee en 10-15 secondes avec le schema complet (230 tables, triggers, vues, procedures).

NouveauArchitecture multi-base

5 BDD de service creees : animgest_template (source pour le clonage), animgest_dev, animgest_demo, animgest_preprod, animgest_test. Workflow de promotion entre ces bases en preparation.

NouveauBouton "Provisionner" et badges UI

Sur la liste des licences, badge orange "A provisionner" sous le tenant_id avant action, passe en vert "Provisionne" apres le click. Le bouton n'apparait que si la licence est active et le tenant pas encore provisionne.

NouveauAudit et tracabilite

Chaque provisioning trace dans animgest_central.billing_events avec event_type=TENANT_PROVISIONED + payload JSON (admin, db_name, source). Plus logs NestJS [AUDIT] TENANT_PROVISIONED admin=... tenant=... db=....

FixAudit log admin=unknown

Les audit logs (LICENSE_CREATED, REVOKED, RENEWED) affichaient admin=unknown car le JWT de session ne contenait que usrId et role. Enrichissement du payload avec login et email apres reconnexion.

Ameliorations techniques

TechContainer API enrichi

Le container micro_api (Node.js Alpine) integre desormais mariadb-client via apk add. Permet l'execution de mariadb-dump | mariadb en pipe shell depuis le service Node pour cloner les schemas BDD.

TechJOIN tenants_registry

Le SELECT de la liste licences embarque LEFT JOIN animgest_central.tenants_registry pour exposer en une seule requete le statut de provisioning (3 colonnes ajoutees : tenant_status, tenant_db_name, tenant_provisioned_at).

TechSentinelles d'environnement

Documentation de R-ENVFILE-1 : tous les fichiers .env necessaires doivent etre listes dans env_file du service Docker. Verifier via docker exec env.

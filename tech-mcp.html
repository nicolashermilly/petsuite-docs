MCP — Model Context Protocol

Ouvrir Anim'Gest aux clients d'IA externes (Claude Desktop, Claude Web, intégrations partenaires) en gardant le contrôle total sur les données, les droits et les règles métier. Architecture pensée pour la confidentialité, l'auditabilité et la conformité RGPD.

🔐 5 couches de sécurité 🛡️ Multi-tenant natif 📋 Audit RGPD-safe ⚡ Stateless & scalable

Qu'est-ce qu'un serveur MCP ?

01

Le Model Context Protocol est un standard ouvert qui permet à un assistant IA (comme Claude) d'accéder à des données et d'effectuer des actions dans des applications tierces de manière contrôlée. Anim'Gest expose un serveur MCP dédié, séparé de l'interface utilisateur classique, qui sert de passerelle sécurisée entre clients IA externes et le système d'information.

Différence avec Eliot

Eliot est l'agent IA intégré dans Anim'Gest — l'utilisateur interagit avec lui depuis l'interface du logiciel pour automatiser ses tâches courantes (créer un devis, programmer une séance, etc.).

Le serveur MCP est un canal complémentaire pour les clients IA externes. Il permet par exemple à un cabinet d'utiliser Claude Desktop directement sur son poste pour interagir avec ses données Anim'Gest, sans passer par l'interface web. Les deux canaux respectent strictement les mêmes règles métier — pas de duplication de logique, pas de risque d'incohérence.

Cas d'usage concrets

02
💼

Cabinets sophistiqués

Un vétérinaire utilise Claude Desktop pour générer ses comptes rendus depuis ses notes vocales, puis pousse directement le résumé dans la fiche animal Anim'Gest.

🤝

Intégrations partenaires

Un éditeur tiers (CRM, comptabilité) consomme les données Anim'Gest via MCP au lieu de coder une intégration REST custom — gain de temps des deux côtés.

🚀

Automatisations CI/CD

Un pipeline de migration ou de test peut interagir avec Anim'Gest comme avec n'importe quelle autre API, mais avec validation et audit renforcés.

📊

Reporting consolidé

Une chaîne de cabinets exploite des analyses cross-sites pour suivre des KPIs agrégés, sans accès aux données individuelles client.

Architecture haut niveau

03

Le serveur MCP est un service indépendant, isolé du reste du système Anim'Gest. Il ne dispose pas d'accès direct aux interfaces utilisateurs ni à la base de données métier en écriture — chaque demande passe par les mêmes routes API que celles utilisées par l'interface web.

Schéma 1 — Architecture haut niveau. Le serveur MCP n'est jamais exposé directement à Internet : tout passe par le reverse proxy avec authentification obligatoire. Côté serveur, le MCP délègue les opérations à l'API métier qui applique les règles d'Anim'Gest.

Modes d'accès aux données

04

Le serveur MCP propose deux modes d'accès, choisis au moment de la conception de chaque outil et jamais modifiables au runtime. Le mode par défaut est volontairement restrictif : tout passe par l'API métier.

Mode par défaut Lecture & écriture

Via l'API métier

Le serveur MCP appelle l'API REST interne d'Anim'Gest, exactement comme le ferait l'interface web. Ce mode garantit que toutes les règles métier sont automatiquement appliquées : validations, calculs de TVA, tarification, workflow de validation des devis, journal comptable, etc.

  • Triggers de cohérence respectés
  • Filtres d'archivage appliqués
  • Validations automatiques en cascade
  • Workflow conforme à l'interface utilisateur
Mode analytique Lecture seule stricte

Accès direct en lecture

Pour des besoins d'analyse ou de reporting cross-tables, certains outils peuvent lire directement la base de données. Cet accès est strictement en lecture, avec un compte dédié dépourvu de tout droit d'écriture, plus une validation applicative qui rejette toute tentative de modification.

  • Compte de lecture sans privilèges d'écriture
  • Validation applicative anti-écriture
  • Limite stricte du nombre de lignes retournées
  • Aucune table sensible accessible

Cycle d'un appel

05

Toute action déclenchée par un client IA externe suit le même chemin : authentification, résolution de contexte, validation des droits, validation des arguments, exécution déléguée, audit. Aucune étape n'est court-circuitée.

Schéma 2 — Cycle complet d'un appel. Les sept étapes sont enchaînées en fail-fast : si l'une échoue, les suivantes ne sont jamais exécutées et l'événement est tracé.

Pipeline d'autorisation

06

L'autorisation s'effectue en quatre niveaux successifs, chacun pouvant refuser l'opération avec un code d'erreur explicite. Chaque niveau est indépendant des autres : un attaquant devrait franchir les quatre simultanément, ce qui est rendu très difficile par construction.

Schéma 3 — Quatre niveaux successifs de validation. Chaque niveau est documenté et journalisé : en cas de refus, la cause est connue, traçable et expliquable au client.

Ce qui est autorisé · ce qui ne l'est pas

07

Les règles de sécurité sont implémentées par construction, pas par convention. Un client IA externe, même malveillant, ne peut techniquement pas contourner ces règles : elles sont appliquées à plusieurs niveaux (permissions de la base, code applicatif, registre des outils).

✓ Autorisé

  • Recherche, lecture et listes (clients, animaux, devis, factures, agenda, etc.)
  • Création de nouveaux enregistrements (devis, séances, paiements, animaux)
  • Modification de statut (valider un devis, marquer une facture payée)
  • Génération de PDF (devis, factures, documents)
  • Envoi d'emails via les canaux configurés du compte
  • Lecture de statistiques agrégées sur le compte

✗ Strictement interdit

  • Suppression d'enregistrement, sous toute forme, sans exception
  • Modification directe de la base (toute opération d'écriture brute)
  • Modification de la structure des données (création/suppression de tables, colonnes)
  • Création ou modification d'utilisateurs ou de comptes
  • Accès aux données d'un autre cabinet que le sien
  • Contournement des consentements RGPD

⚠ Pourquoi ces interdictions sont absolues

Les opérations destructives ou structurelles présentent des risques de perte de données ou de cohérence métier qui ne peuvent pas être validés correctement par une IA, même sophistiquée. Plutôt que de tenter de filtrer au cas par cas, nous interdisons par principe ces opérations au niveau du serveur MCP. Les utilisateurs gardent évidemment toutes ces possibilités dans l'interface web classique d'Anim'Gest, qui dispose des garde-fous interactifs nécessaires (confirmations, rollbacks, journaux, etc.).

Défense en profondeur

08

La sécurité du serveur MCP repose sur cinq couches indépendantes. Chaque couche est suffisante pour bloquer une catégorie d'attaques, mais aucune n'est seule responsable de la sécurité du système. Pour qu'un attaquant atteigne les données, il doit franchir successivement les cinq couches.

Schéma 4 — Cinq couches concentriques de sécurité. Le principe est qu'aucune couche ne peut être contournée par les autres. Une vulnérabilité dans une couche n'expose pas le système si les autres couches restent intactes.

Audit & traçabilité RGPD

09

Chaque appel d'outil est tracé dans un journal d'audit centralisé. Le contenu du journal respecte par défaut le principe de minimisation RGPD : aucune donnée personnelle n'est stockée en clair.

Ce qui est tracé pour chaque appel

  • Quel outil a été appelé (ex : create_devis)
  • Quel cabinet et quel utilisateur ont effectué l'appel
  • Date et heure précises
  • Statut : succès, refus ou erreur
  • Latence d'exécution en millisecondes
  • Empreinte cryptographique des arguments (pas leur contenu)
  • Code et message d'erreur le cas échéant

Pourquoi une empreinte plutôt que le contenu

Stocker les arguments en clair impliquerait de garder potentiellement des données personnelles dans les journaux. À la place, nous stockons une empreinte cryptographique (hash SHA-256) des arguments. Cette empreinte permet :

  • De détecter des patterns d'usage (même argument répété plusieurs fois)
  • De prouver juridiquement qu'un appel a bien eu lieu (auditabilité)
  • De ne jamais exposer de données client en cas d'audit ou de fuite
  • De respecter le principe de minimisation imposé par le RGPD

Performance & disponibilité

10

Le serveur MCP est conçu pour servir plusieurs cabinets simultanément avec une latence perçue identique à celle de l'interface web classique. Il est stateless : aucune donnée n'est conservée entre les requêtes, ce qui simplifie le redémarrage et le passage à l'échelle.

Type d'opérationLatence cible (95e centile)Comportement en cas d'incident
Recherche / lecture simplemoins de 100 msRéessai automatique côté client MCP
Création / modificationmoins de 300 msErreur explicite + journal d'audit
Analytique multi-tablesmoins de 500 msLimite dure du nombre de lignes
Validation droitsmoins de 50 msCache local avec invalidation

Disponibilité indépendante de l'interface utilisateur

Une indisponibilité du serveur MCP n'affecte ni l'interface web ni les sauvegardes ni les emails automatiques. Réciproquement, une interruption de l'interface web n'empêche pas l'API métier (et donc le serveur MCP) de continuer à fonctionner. Cette indépendance est volontaire et améliore la robustesse globale du système.

Et pour la suite ?

11

Le serveur MCP est mis en place progressivement, par phases :

🌱

Phase 1 — Lecture seule

Un client IA peut consulter ses données : clients, animaux, devis, factures. Aucune action d'écriture autorisée durant cette phase.

📝

Phase 2 — Création basique

Création de devis, séances, animaux, paiements. Triggers métier appliqués comme dans l'interface utilisateur classique.

🤝

Phase 3 — Multi-cabinet

Vue consolidée pour les chaînes de cabinets, avec respect strict des consentements RGPD du propriétaire de l'animal.

🚀

Phase 4 — Intégrations

Ouverture aux partenaires externes, écosystème d'outils et flux automatisés certifiés.

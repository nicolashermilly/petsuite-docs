Pipeline de déploiement production
# .github/workflows/deploy-prod.yml name: Deploy Production on: push: tags: ['v*.*.*'] jobs: build-and-push: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Login GHCR uses: docker/login-action@v3 with: registry: ghcr.io username: ${{ github.actor }} password: ${{ secrets.GITHUB_TOKEN }} - name: Build & push API uses: docker/build-push-action@v5 with: context: ./api push: true tags: ghcr.io/nicolashermilly/petsuite-api:${{ github.ref_name }} - name: Build & push Frontend uses: docker/build-push-action@v5 with: context: ./frontend push: true tags: ghcr.io/nicolashermilly/petsuite-frontend:${{ github.ref_name }} deploy: needs: build-and-push runs-on: ubuntu-latest steps: - name: Deploy via SSH uses: appleboy/ssh-action@v1 with: host: ${{ secrets.PROD_HOST }} username: deploy key: ${{ secrets.PROD_SSH_KEY }} script: | cd /opt/petsuite export TAG=${{ github.ref_name }} docker compose -f docker-compose.prod.yml pull docker compose -f docker-compose.prod.yml up -d --no-deps api frontend worker docker image prune -f echo "Deployed $TAG at $(date)"
Le déploiement utilise --no-deps pour ne redémarrer que les conteneurs mis à jour. MariaDB et Redis ne sont jamais redémarrés par le pipeline.

08Backup & restauration
💾
Stratégie backup BDD
FréquenceQuotidien à 02h00 UTC (cron GitHub Actions)
Méthodemysqldump par schema tenant + compression gzip
ChiffrementGPG AES-256 avant upload S3
DestinationBucket S3 dédié backups (région différente)
Rétention7 jours quotidiens · 4 semaines hebdomadaires · 12 mois mensuels
Test restaurationMensuel automatique sur environnement de test
🔁
Script backup (cron)
#!/bin/bash — backup-all-tenants.sh DATE=$(date +%Y%m%d_%H%M) BACKUP_DIR=/tmp/backups/$DATE mkdir -p $BACKUP_DIR # Lister tous les schemas tenants TENANTS=$(mysql -u @lgorithme -p$DB_PASS \ -e "SHOW DATABASES LIKE 'micro_logiciel_%'" \ -s --skip-column-names) for DB in $TENANTS; do mysqldump -u @lgorithme -p$DB_PASS \ --single-transaction --quick \ $DB | gzip | gpg --symmetric \ --cipher-algo AES256 \ -o $BACKUP_DIR/$DB.sql.gz.gpg done # Upload vers S3 aws s3 sync $BACKUP_DIR \ s3://petsuite-backups/$DATE/ \ --storage-class STANDARD_IA rm -rf $BACKUP_DIR echo "Backup $DATE terminé — $(ls $BACKUP_DIR | wc -l) tenants"
Procédure de restauration d'urgence
1
Identifier le backup
aws s3 ls s3://petsuite-backups/ → choisir la date/heure du backup souhaité.
2
Télécharger et déchiffrer
aws s3 cp s3://petsuite-backups/{date}/{schema}.sql.gz.gpg . → gpg --decrypt → gunzip.
3
Restaurer le schema
mysql -u @lgorithme -p {schema} < {schema}.sql — Sur environnement isolé d'abord pour vérification.
4
Vérifier et basculer
Contrôle intégrité des données → mise à jour tenant_shard si basculement de serveur → notification client.
RTO (Recovery Time Objective) cible : < 4h. RPO (Recovery Point Objective) cible : < 24h (backup quotidien). Pour un SLA plus exigeant : binlog MariaDB en temps réel.

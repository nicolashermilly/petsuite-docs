🏗️ Architecture générale

La GED (Gestion Électronique de Documents) d'Anim'Gest repose sur un stockage filesystem local sur le serveur, référencé en base de données. Les fichiers sont organisés par type de document et stockés dans un répertoire dédié monté comme volume Docker.

ℹ️
Le chemin racine GED est configurable via param_entreprise.ENT_GED_PATH. Toutes les routes doivent lire ce paramètre dynamiquement — jamais hardcoder le chemin.

📂 Chemin racine (prod)

C:\AnimGest_GED\
Monté dans Docker comme volume bind-mount sur /app/ged/

🐳 Volume Docker

Défini dans docker-compose.yml sous le service micro_api — permet la persistance des fichiers entre rebuilds.

🌲 Arborescence filesystem

Les fichiers sont organisés par module fonctionnel, chaque sous-dossier correspondant à un type de document GED.

C:\AnimGest_GED\ ← ENT_GED_PATH ├── editions\ ← Devis PDF, Factures PDF générées par Puppeteer │ ├── DEV-2604-0001.pdf │ └── FAC-2604-0042.pdf ├── widgets\ ← Logos, images widget IA publique │ └── logo_animgest.svg ├── animaux\ ← Photos d'animaux, carnet de santé │ ├── ani_22_frimousse.jpg │ └── ani_22_passeport.pdf ├── seances\ ← CR de séance, EDC signés │ └── sea_157_cr.pdf ├── devis\ ← Devis signés Yousign │ └── DEV-2604-0001_signe.pdf ├── factures\ ← Factures envoyées par email │ └── FAC-2604-0042_envoyee.pdf ├── clients\ ← Documents client (CGV signées, etc.) └── tmp\ ← Fichiers temporaires (purge auto 24h)
⚠️
Le dossier tmp\ est purgé automatiquement toutes les 24h. Ne jamais stocker de fichiers définitifs dans tmp\.

🗄️ Table ged_fichier

Référentiel central de tous les fichiers GED. Chaque ligne référence un fichier physique sur le filesystem et son contexte métier.

CREATE TABLE ged_fichier ( GF_ID INT AUTO_INCREMENT PRIMARY KEY, GF_MODULE VARCHAR(50) NOT NULL COMMENT 'Editions|Widgets|Animaux|Seances|Devis|Factures|Clients', GF_REF_ID INT NULL COMMENT 'ID de l objet lie (ANI_ID, SEA_ID, DCE_ID…)', GF_NOM VARCHAR(255) NOT NULL COMMENT 'Nom du fichier affiché', GF_CHEMIN VARCHAR(500) NOT NULL COMMENT 'Chemin relatif depuis ENT_GED_PATH', GF_TYPE_MIME VARCHAR(100) NULL COMMENT 'application/pdf | image/jpeg | …', GF_TAILLE INT NULL COMMENT 'Taille en octets', GF_HASH VARCHAR(64) NULL COMMENT 'SHA-256 pour détection doublon', ENT_ID INT NOT NULL DEFAULT 1 COMMENT 'Entreprise propriétaire', GF_CREATED_AT DATETIME NOT NULL DEFAULT NOW(), GF_UPDATED_AT DATETIME NULL ON UPDATE NOW() ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

🐳 Volume Docker

Le dossier GED est monté comme volume bind-mount dans le service micro_api afin de persister les fichiers entre les rebuilds de l'image.

# docker-compose.yml — extrait service micro_api services: micro_api: volumes: - C:\AnimGest_GED:/app/ged # bind-mount Windows → Linux container - ./api:/app/src # source code environment: - GED_PATH=/app/ged # lu par les routes upload

🖥️ Côté Windows (host)

Chemin physique : C:\AnimGest_GED\
Configurable via param_entreprise.ENT_GED_PATH en base.

🐧 Côté container Linux

Chemin interne : /app/ged/
Accessible par NestJS via la variable d'env GED_PATH.

⚠️
Ne jamais utiliser docker compose down -v — cela supprimerait les volumes nommés. Seul docker compose down (sans -v) est sûr pour les données GED.

⬆️ Mécanique d'upload

L'upload de fichiers dans la GED suit un pattern standardisé : le fichier est écrit sur le filesystem puis référencé en base dans ged_fichier.

// Pattern NestJS — route upload GED (custom-routes.ts) instance.use('/ged/upload', async (req: any, res: any) => { if (req.method !== 'POST') return next(); // 1. Lire chemin depuis param_entreprise const db = await getDb(); const [params] = await (db as any).execute( `SELECT ENT_GED_PATH FROM param_entreprise WHERE ENT_ID = 1` ); const gedPath = params[0]?.ENT_GED_PATH ?? '/app/ged'; // 2. Écrire fichier dans gedPath/module/ const { module, ref_id, filename, content } = await _readBody(req); const filePath = path.join(gedPath, module, filename); fs.writeFileSync(filePath, Buffer.from(content, 'base64')); // 3. Insérer référence en BDD await (db as any).execute( `INSERT INTO ged_fichier (GF_MODULE, GF_REF_ID, GF_NOM, GF_CHEMIN, ENT_ID) VALUES (?, ?, ?, ?, 1)`, [module, ref_id, filename, `${module}/${filename}`] ); res.json({ ok: true, path: filePath }); });
ℹ️
Les PDF générés par Puppeteer (devis, factures) utilisent ce même pattern — ils sont écrits dans editions/ puis référencés dans ged_fichier avec GF_MODULE = 'Editions'.

⚠️ Règles critiques

🔴 ENT_GED_PATH dynamique

Toujours lire param_entreprise.ENT_GED_PATH — jamais hardcoder C:\AnimGest_GED dans le code source.

🔴 Chemin relatif en BDD

GF_CHEMIN stocke un chemin relatif depuis ENT_GED_PATH. Jamais le chemin absolu complet.

🟠 Volume Docker persistant

Ne jamais recréer le container avec -v. Le bind-mount C:\AnimGest_GED:/app/ged doit rester intact entre rebuilds.

🟠 BDD GED séparée obsolète

La base micro_logiciel_ged est en cours de suppression (S67+). La GED est maintenant gérée via micro_logiciel.ged_fichier.

🟡 PDF via modal uniquement

Les PDF GED s'ouvrent dans une modale iframe — jamais window.open() ni navigation vers une nouvelle page.

🟡 Purge tmp/ automatique

Le dossier tmp/ est purgé automatiquement. Les fichiers définitifs doivent être déplacés dans leur module avant 24h.

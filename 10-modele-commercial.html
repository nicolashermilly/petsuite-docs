03Workflow de souscription
📋
Étapes du parcours
1
Formulaire prospect
Saisie : raison sociale, SIRET, email admin, sous-domaine souhaité, plan choisi. Vérification SIRET format + unicité.
Manuel
2
Affichage & acceptation CGV
Présentation de la version CGV en vigueur. Checkbox obligatoire + horodatage serveur. IP client enregistrée.
Manuel Légal
3
Saisie IBAN + signature mandat SEPA
Formulaire IBAN/BIC/titulaire. Envoi à GoCardless ou Stripe pour création du mandat. Référence mandat retournée et stockée.
Manuel Système
4
Création contrat en base
INSERT dans contrat (snapshot CGV, dates, montant figé). Génération numéro CTR-YYMM-0001. tenant_id créé.
Auto
5
Provisionnement automatique tenant
Création schema MariaDB + migration Flyway. Création bucket S3 GED. Configuration DNS sous-domaine + SSL Let's Encrypt. Enregistrement dans tenant_shard.
Auto Système
6
Création compte admin société
Insertion utilisateur rôle ADMIN dans le schema tenant. Email de bienvenue avec lien de première connexion (48h).
Auto
7
Premier prélèvement J+1
Prélèvement soumis le lendemain de la souscription. Statut tenant → ACTIF dès confirmation bancaire (délai 2–5 jours ouvrés SEPA).
Auto Système
🔄
Cycle de prélèvement mensuel
J-3
Email de notification
Email automatique au client : montant à venir, date, IBAN utilisé (4 derniers chiffres).
Auto
J
Soumission prélèvement
INSERT prelevement statut PLANIFIE → appel API GoCardless/Stripe → statut SOUMIS.
Auto
J+2 à J+5
Webhook confirmation
Réception webhook → statut OK ou ECHOUE. Mise à jour prelevement + audit_plateforme.
Système
📅
Calendrier engagement
J0 — SouscriptionContrat créé, provisionnement lancé
J0 à M+6Période d'engagement — résiliation impossible
M+6Fin engagement — résiliable à tout moment
Résiliation + 1 moisDernier prélèvement — accès lecture seule
Résiliation + 3 moisFermeture accès — export disponible sur demande
Résiliation + 5 ansPurge automatique (script planifié)
04Intégration SEPA
Comparatif GoCardless vs Stripe SEPA
Recommandé
GoCardless
Spécialiste prélèvements récurrents
Spécialité100% dédié aux prélèvements SEPA récurrents
Frais~1% + 0,20€ / transaction — plafond négociable
Mandat SEPA B2BNatif — délais de remboursement réduits (B2B)
WebhooksTrès complets — statuts granulaires (paid, failed, cancelled…)
DashboardInterface dédiée gestion mandats + exports
IntégrationAPI REST + bibliothèques Node.js officielles
InconvénientPas de gestion carte bancaire — SEPA uniquement
Stripe SEPA Debit
Solution tout-en-un
SpécialitéMulti-moyens de paiement (CB + SEPA + virement)
Frais0,8% plafonné à 6€ / transaction
Mandat SEPAVia SetupIntent — processus légèrement plus complexe
WebhooksBons — event payment_intent.succeeded/failed
DashboardDashboard Stripe complet — déjà utilisé pour portail client
IntégrationSDK Node.js mature — déjà intégré dans le projet
AvantageUnification avec Stripe portail client existant
Recommandation : GoCardless pour la plateforme SaaS (spécialisé, frais compétitifs à volume, SEPA B2B natif). Stripe reste utilisé pour le portail client (paiement CB one-shot). Les deux peuvent coexister.
Flux technique mandat & prélèvement (GoCardless)
// 1. Création mandat à la souscription POST https://api.gocardless.com/redirect_flows { "session_token": "uuid_session", "success_redirect_url": "https://app.petsuite.fr/souscription/confirmation", "prefilled_customer": { "email": "[email protected]", "company_name": "Cabinet Martin" } } // → redirect_url renvoyée → client saisit IBAN sur page GoCardless // 2. Confirmation (webhook redirect_flows/completed) POST https://api.gocardless.com/redirect_flows/:id/actions/complete // → mandate_id retourné → stocker dans mandat_sepa.man_provider_id // 3. Prélèvement mensuel (cron J de chaque mois) POST https://api.gocardless.com/payments { "amount": 4900, // en centimes "currency": "EUR", "links": { "mandate": "MD_XXXXXXXX" }, "description": "PetSuite - Abonnement Mars 2026" } // → payment_id → stocker dans prelevement.pre_provider_id // 4. Webhooks à écouter payments.paid_out → statut OK payments.failed → statut ECHOUE + motif + incrémenter tentative_num payments.charged_back → statut REMBOURSE + alerte super admin
Gestion automatique des échecs de prélèvement
1
1er échec — J+1
Webhook payments.failed reçu. Statut prelevement → ECHOUE. Email automatique au client : "Votre prélèvement a échoué — régularisez sous 7 jours." pre_tentative_num = 1. Accès normal maintenu.
Auto
2
Nouvelle tentative — J+8
Si non régularisé sous 7 jours : second prélèvement soumis automatiquement. Email de relance envoyé.
Auto
3
2e échec — suspension
pre_tentative_num = 2. Statut tenant → SUSPENDU. Accès application → lecture seule uniquement. Email client : "Accès suspendu — mettez à jour votre IBAN ou régularisez." Alerte super admin.
Auto Légal
4
Réactivation
Client saisit nouveau mandat SEPA OU règle l'arriéré par virement. Super admin valide manuellement → statut tenant → ACTIF. Nouveau prélèvement planifié.
Manuel
5
Sans régularisation J+30
Mise en demeure par email. Résiliation d'office si l'engagement de 6 mois est atteint. Dans l'engagement : suspension maintenue jusqu'à la fin des 6 mois.
Légal

07Provisionnement automatique tenant
⚡
Séquence de création
1
Génération tenant_id
Sequential : tenant_0001, tenant_0002… Insertion dans table tenant (statut PROVISIONNEMENT).
2
Création schema MariaDB
CREATE DATABASE micro_logiciel_{tenant_id}. Application migrations Flyway. Insertion dans tenant_shard.
3
Création bucket S3 GED
Bucket petsuite-ged-{tenant_id}. Politique cycle de vie hot/warm/cold. Chiffrement SSE-S3.
4
DNS + SSL
{sous_domaine}.petsuite.fr → reverse proxy. Certificat Let's Encrypt auto via Traefik.
5
Compte admin société
INSERT utilisateur (email_admin, rôle ADMIN) dans le schema tenant. Email bienvenue avec lien 48h.
6
Statut → ACTIF
Une fois le premier prélèvement confirmé par webhook : tenant.ten_statut → ACTIF.
💻
Commandes CLI
# Créer un nouveau tenant node tenant-cli.js create \ --id=tenant_0042 \ --siret=12345678901234 \ --admin=[email protected] \ --plan=CABINET # Vérifier l'état node tenant-cli.js status tenant_0042 # Appliquer migrations (MAJ produit) node tenant-cli.js migrate-all # Suspendre node tenant-cli.js suspend tenant_0042 # Archiver (J+3 mois post-résiliation) node tenant-cli.js archive tenant_0042 # Purge (J+5 ans — IRRÉVERSIBLE) node tenant-cli.js purge tenant_0042 --confirm
La commande purge est irréversible. Vérifier ten_date_purge_prevue avant exécution. Logger l'action dans audit_plateforme.
08Gestion des CGV versionnées
📄
Principe
Snapshot immuableTexte complet CGV copié dans contrat.con_cgv_texte_snapshot à la signature — ne change jamais
Hash SHA-256Empreinte calculée sur le texte — preuve d'intégrité
Nouvelle versionCréer une nouvelle ligne cgv_version — pas de modification de l'existante
Version activeUn seul enregistrement avec cgv_actuelle = 1
📬
Processus de mise à jour CGV
1
Rédaction nouvelle version
INSERT cgv_version (cgv_actuelle = 0). Hash SHA-256 calculé. Date publication future définie.
2
Notification J-30
Email à tous les tenants ACTIF : "Nos CGV évolueront le [date]." Lien vers nouvelles CGV.
3
Application
À la date de publication : cgv_actuelle = 1 pour la nouvelle, 0 pour l'ancienne.
4
Re-acceptation obligatoire
À la prochaine connexion admin : modal d'acceptation bloquant. Nouveau snapshot enregistré dans contrat.

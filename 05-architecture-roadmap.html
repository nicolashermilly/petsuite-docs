08 Architecture multi-tenant SaaS
Cette architecture est définie pour le passage en production SaaS. L'application tourne actuellement en mode mono-tenant local (Docker).
Couches d'infrastructure
Client — navigateur / tablette / mobile
URL : societe.petsuite.fr
CDN Cloudflare
Assets statiques, vidéos GIF modules
Reverse proxy Nginx / Traefik
Routage par sous-domaine → tenant_id
API Gateway + Auth JWT
Injection tenant_id — zéro fuite inter-société
Shard registry
tenant_id → host + port + schema MariaDB
Schema MariaDB par société
micro_logiciel_{tenant_id} — isolation totale RGPD
Bucket S3 GED par société
/{tenant_id}/{type}/{YYYY}/{MM}/fichier
BDD Plateforme (super admin)
contrats · licences · mandats SEPA · CGV · audit global
Principes clés multi-tenant
IsolationUn schema MariaDB par société — jamais de table avec tenant_id partagé
ScalabilitéAjout de serveurs MariaDB sans toucher aux tenants existants
MigrationsFlyway / Liquibase — appliqué sur tous les schemas à chaque update
CacheRedis — sessions, JWT, droits utilisateur (TTL 15 min)
QueueRabbitMQ / SQS — PDF, emails, prélèvements SEPA
PerformanceRead replicas pour exports et rapports — pas sur primary
DNS autosous-domaine créé + certificat SSL Let's Encrypt au provisionnement
MonitoringPrometheus + Grafana — alertes quota par tenant
